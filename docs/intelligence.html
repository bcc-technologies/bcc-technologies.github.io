What it is

Intelligence is an internal Scientific & Technology Intelligence module inside the admin dashboard. It helps BCC monitor papers, grants, patents, institutions, topics, and derived strategic signals across product lines such as MAP-Nano, MAP-Bio, MAP-Med, and MAP-Ing.

Problem solved

The module replaces scattered manual review with a single radar that helps BCC detect product opportunities, scientific trends, partner candidates, content ideas, grant windows, and early competitive pressure.

Architecture

  1. Supabase stores sources, topics, papers, grants, patents, institutions, signals, runs, and settings.
  2. Connectors fetch and normalize external items.
  3. Sync + scoring deduplicates, stores data, and generates signals.
  4. Admin dashboard lets admins review data and trigger runs.

Manual sync

From the dashboard

Open /admin-dashboard.html#intelligence and use:

  • Run Intelligence Sync
  • Fetch latest papers
  • Generate signals

Fetch grants and Fetch patents are still placeholders.

From the CLI

node scripts/sync-intelligence.mjs --dry-run --source arxiv,openalex,crossref --keyword "SEM image analysis" --limit 10
SUPABASE_URL=... \
SUPABASE_SERVICE_ROLE_KEY=... \
OPENALEX_API_KEY=... \
OPENALEX_EMAIL=you@example.com \
node scripts/sync-intelligence.mjs --source arxiv,openalex,crossref --query "scientific image analysis" --limit 15
OpenAlex free quota

Keep normal per-source limits around 20 to 50, use dry-run for broad exploratory searches, and avoid repeated generic full-text syncs in the same day.

Adding a new connector

Keep the source isolated in its own module. Do not hardcode new fetch logic into one shared function.

{
  sourceName,
  sourceType,
  search(query),
  fetchById?(id)
}
  1. Create a new module under scripts/intelligence/connectors/.
  2. Normalize outbound items with scripts/intelligence/connectors/base.mjs.
  3. Use scripts/intelligence/http.mjs for timeout, retry, and rate-limit behavior.
  4. Return normalized records with rawData.
  5. Register the connector in scripts/intelligence/connectors/index.mjs.
  6. Use environment variables for credentials.
  7. Add tests for normalization and connector behavior.

How scores are calculated

Signals are heuristic, not LLM-generated.

Opportunity Score

0.25 * topicGrowth
+ 0.20 * proximityToBCC
+ 0.15 * fundingPresence
+ 0.15 * technicalPainDetected
+ 0.10 * activeInstitutions
+ 0.10 * competitiveWhiteSpace
+ 0.05 * openDataAvailability

Actionability Score

Estimated from:

  • dataAvailability
  • clarityOfUseCase
  • easeOfContact
  • compatibilityWithCurrentProduct
  • contentPotential

Current heuristics

  • proximityToBCC: keyword matching against MAP lines and topic evidence.
  • technicalPainDetected: terms such as manual, bottleneck, time-consuming, segmentation challenge, annotation, thresholding, low accuracy, noisy images.
  • fundingPresence: related grants increase the score.
  • activeInstitutions: distinct institutions across matching papers and grants.
  • openDataAvailability: boosted by open access or dataset-like evidence.
Evidence rule

Signals are not emitted without evidenceRefs.

Deduplication and normalization

Exact merge keys for papers:

  • DOI
  • arXiv ID
  • sourceType + externalId

Near-identical normalized titles do not auto-merge ambiguous records. They are marked as possible duplicates so the system stays conservative.

Normalized fields currently include titles, authors, institutions, dates, source URLs, topics, and keywords. rawData is always preserved.

Security model

  • Only admins can access Intelligence admin routes.
  • API keys and backend secrets are not exposed to the frontend.
  • The manual sync Edge Function is admin-only.
  • Manual sync has basic rate limiting and active-run protection.
  • External requests use timeout and moderate retries.
  • Error logging redacts sensitive token-like material when possible.
  • External links in the UI are sanitized and restricted to safe protocols.

Testing notes

Current limitations

  • Fetch grants and Fetch patents are not implemented end-to-end yet.
  • The current sync is strongest on papers.
  • Signal scoring is intentionally simple and heuristic.
  • Duplicate review exists as metadata, not yet as a full analyst workflow.
  • Institution enrichment is still lightweight.

What remains pending

  • real grants connectors
  • real patents connectors
  • duplicate review workflow in the dashboard
  • better institution/entity resolution
  • analyst notes and richer signal lifecycle
  • saved views and exports
  • richer ranking once simple heuristics stop being enough
